# TowerRandomDefense

### 1. ___🔧프로젝트 소개___ <br/>
랜덤 영웅 타워 디펜스는  무작위로 생성되는 영웅 타워를 전략적으로 배치, 강화하여 몰려오는 적을 막아내는 2D 디펜스 게임입니다.

### 2. ___💡개발 목적 및 학습 목표___ <br/>
1. 외부 데이터 관리(GoogleSheet 연동을 통한 밸런싱 시스템 구축) <br/>
2. 객체 풀링 기반의 메모리 최적화 <br/>
3. ScriptableObject 기반의 데이터 아키텍쳐 설계 <br/>
4. 클래스 구조화 및 상속을 통한 확장성 확보 <br/>

### 3. ___🧠핵심구현___ <br/>
1) ____Google Sheet와 SO를 분리한 데이터 관리 및 메모리 최적화 구조:____ <br/>
# 1. 문제 상황

초기에는 타워의 기본 스탯은 구글 시트로, 강화 시스템은 SO로 분리하여 데이터 관리를 했습니다.

그러나 이 구조는 타워 인스턴스가 동일한 구글 시트 데이터를 반복적으로 접근하게 되어 메모리 및 데이터 관리 측면에서 비효율적이라 판단 했습니다.

하지만 구글시트만 사용할 경우 협업에 좋고 대량의 데이터 관리에 있어서 용이하지만 메모리 문제가 있으며, SO만 사용할 경우 확장성과 밸런스 조정 측면에서 불편함이 있었습니다.

---

# 2. 해결 방안

이에 구글 시트를 외부 데이터 관리 도구로 사용하고, SO를 런타임 데이터 컨테이너 구조를 설계했습니다. 데이터 수정은 구글 시트에서 유연하게 처리하고, 런타임에서는 안정적인 SO 참조 구조를 유지 하여 메모리 중복을 줄이고 데이터 확장성을 확보했습니다.

2) ____Factory 패턴+ ObjectPool :____ <br/>
# 1. 문제 상황

타워, 투사체, 몬스터, 데미지 텍스트 처럼 생성과 제거가 반복되는 여러 종류의 오브젝트를 다루기 때문에 런타임 성능을 고려해 ObjectPool을 먼저 적용 했습니다. 

하지만 ObjectPool만 사용한 구조에서는 각 시스템이 직접 어떤 풀을 사용할지 판단해야 했고, 곳곳에 타입 분기와 풀 선택 로직이 흩어지기 시작했습니다.

이로 인해 오브젝트 종류가 늘어날수록 새로운 타입을 추가할 때마다 반복 수정해야 했고, 실수로 인한 런타임 오류 가능성이 증가했습니다.

# 2. 해결 방안

![image.png](attachment:a140cd6c-893f-4adc-a48b-c33cd64bc9a9:image.png)

ObjectPool, TowerFactory, TowerSpawner 모두 설치 흐름에 관여하지만, 각 클래스의 책임이 섞일 경우 유지보수가 어렵다고 판단했습니다.

Spawner: “설치 요청”

Factory: “타입별 생성 결정”

ObjectPool: “인스턴스 재사용”

명확한 역할 분리하여 시퀀스를 구성하였습니다.

ObjectPool은 오브젝트의 재 사용만 담당하고, “어떤 타입의 오브젝트를 어떤 방식으로 생성할 지”에 대한 규칙은 Factory 패턴으로 분리하여 설계 하였습니다.

![image.png](attachment:8bbc2359-d082-4334-8ecb-c2e014197329:image.png)

외부에서는 타워 타입과 위치만 전달하고 Factory 내부에서 타입에 맞는 Pool을 선택하고 인스턴스를 반환함으로써 문제를 해결했습니다.

![image.png](attachment:2be9f5e0-5554-4f05-bda6-f9225b22334d:image.png)

또한 ObjectPool은 제네릭으로 구현해 컴파일 타임에 타입이 보장되어 태그나 캐스팅에 의존하던 방식보다 안정적인 구조를 유지했습니다.

이 구조를 통해 오브젝트 재 사용에 따른 성능 이점과 타입 확장 시 수정 범위를 Factory만으로 제한할 수 있는 확장성 있는 생성 구조를 만들 수 있었습니다.

# 1. 오브젝트 풀 기반 성능 최적화

### 반복 생성 오브젝트를 풀링하여 모바일 환경에서 GC 관

**문제/원인**

디펜스 게임 특성상 몬스터, 타워, 프로젝타일, 데미지 텍스트 등이 전투 중 반복적으로 생성/제거됩니다. 모바일 환경에서는 잦은 Instantiate/Destroy 호출이 GC를 유발하여 프레임 드랍과 끊김 현상을 발생 리스크가 있습니다.

**설계/해결**

전투 중 생성 빈도가 높은 모든 주요 오브젝트를 오브젝트 풀링 구조로 통합 관리했습니다. 생성 대신 재사용 방식을 적용하여 런타임 메모리 할당/해제를 최소화했습니다.

**결과/효과**

GC낭비를 최소화하고 프레임 안정성을 확보하여, 웨이브가 길어져도 끊김 없는 안정적인 플레이 환경을 유지할 수 있도록 최적화 했습니다.

---

# 2. 데이터 기반 타워 강화 시스템 설계

### Google Sheets + ScriptableObject 캐싱을 통한 밸런싱 유연성 확보

**문제/원인**

타워 레벨 상승에 따른 능력치 증가를 코드에 직접 작성할 경우, 밸런스 조정 시 코드 수정과 재빌드가 반복적으로 필요해 유지 보수단계에서 비효율적입니다.

**설계/해결**

타워 타입/레벨별 능력치를 외부 API를 연동한 Google Sheets에서 관리하고, 빌드 시 ScriptableObject로 캐싱하는 데이터 파이프라인을 구축했습니다. 강화 시 고정 증가 방식이 아닌 데이터 참조 기반 구조를 적용했습니다.

**결과/효과**

밸런스 조정이 필요할 경우 코드 수정 없이 데이터 수정만으로 대응할 수 있어 확장성과 유지보수성이 향상되었으며, 런타임에서는 SO 캐싱으로 안정적인 데이터 접근을 유지했습니다.

---

# 3. 웨이브 시스템 설계 및 메모리 관리 전략

### 데이터 기반 웨이브 구조와 레벨별 풀 릴리즈

**문제/원인**

웨이브를 코드에 하드 코딩할 경우 기획 변경 대응이 어렵고, 웨이브 진행 중 누적되는 오브젝트로 인해 메모리 사용량이 증가할 수 있습니다.

**설계/해결**

웨이브 패턴을 데이터 기반 구조로 설계하여 스폰 수, 간격, 난이도 등을 외부에서 조정 가능하도록 구성했습니다. 또한 웨이브 흐름을 ‘스폰 → 생존 몬스터 카운트 → 다음 웨이브 타이머 → 레벨 증가’ 구조로 단순화했습니다. 이전 웨이브 몬스터는 앞으로 재사용하지 않기에, 레벨 상승 시 이전 레벨 풀을 릴리즈하여 메모리를 회수하도록 설계했습니다.

**결과/효과**

기획 변경에 유연하게 대응할 수 있는 웨이브 시스템을 구축했으며, 메모리 사용량을 제어해 장시간 플레이 시에도 안정적인 성능을 유지했습니다.

# 4. 조작 UX 개선 시스템 설계

### 선택/피드백 시각화를 통해 모바일 조작 실수 최소화

**문제/원인**

디펜스 게임은 배치와 이동이 핵심이지만, 모바일 환경에서는 터치 오입력으로 인해 불쾌한 경험을 제공할 수 있습니다.

**설계/해결**

타워 선택 시 공격 사거리를 시각화 하여 사거리에 따른 전략적으로 타워를 배치할 수 있으며,  타일 선택 시 하이라이트 피드백을 제공하여 현재 상태를 명확히 표시했습니다. 타워 이동은 ‘타워 선택 → 타일 1클릭’ 구조로 단순화해 조작 단계를 최소화했습니다.

**결과/효과**

사용자의 조작 실수를 줄이고 전략 판단을 직관적으로 지원하여 모바일 환경에 적합한 UX를 구현했습니다.
